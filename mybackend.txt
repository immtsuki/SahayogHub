## 🧠 Detailed Backend Architecture

SahayogHub backend is designed as a secure, scalable, and modular civic-tech system built using **Django**, **Django REST Framework**, and **PostgreSQL**. The backend powers all core services including Lost & Found, Missing Persons, Vehicle Search, Emergency SOS, Fundraising, Road Hazard Reporting, Corruption Reporting, and secure identity/document-based verification.

The system is divided into two major service modes:

```txt
1. Find & Report System
   → Used to find people, vehicles, pets, documents, and lost objects.

2. Secure Document & Identity System
   → Used to store sensitive identity references such as citizenship number,
     passport number, license number, and verified ownership details.
```

---

## 🔍 1. Find & Report System

The Find & Report system is the public-facing discovery module of SahayogHub. It allows citizens to submit and search reports related to:

* Missing persons
* Lost or found vehicles
* Lost pets
* Lost documents
* Found objects
* Road hazards
* Emergency incidents
* Suspicious or corruption-related reports

### Main Purpose

The goal of this module is to make community reporting searchable, organized, and actionable.

```txt
Citizen submits report
        ↓
Backend validates data
        ↓
Images and details are stored
        ↓
Matching engine checks similar records
        ↓
Possible matches are ranked
        ↓
Users and authorities are notified
```

### Supported Report Types

| Report Type       | Example                                      |
| ----------------- | -------------------------------------------- |
| Missing Person    | Missing child, elderly person, family member |
| Vehicle           | Lost bike, stolen scooter, abandoned vehicle |
| Pet               | Lost dog, found cat                          |
| Object            | Wallet, bag, mobile phone, documents         |
| Document          | Passport, license, citizenship card          |
| Road Hazard       | Broken road, landslide, blocked drainage     |
| Emergency         | SOS request, accident, urgent help           |
| Corruption Report | Bribery, misuse of public service            |

## 🧾 2. Secure Document & Identity System

This module stores sensitive identity-related information in a protected way. It is not fully public and should only be accessible to verified users, admins, or authorized personnel.

Examples of stored document data:

* Citizenship number
* Passport number
* Driving license number
* Vehicle number plate
* National ID number
* Document owner name
* Date of birth
* Issued district
* Contact number
* Verification status

### Important Security Rule

Sensitive numbers such as citizenship number, passport number, and license number should **never be stored as plain public data**.

They should be:

* Encrypted in the database
* Hidden from normal users
* Accessible only through permission-based roles
* Logged whenever viewed or updated
* Used for matching and verification, not public display

Example:

```txt
Public user sees:
Citizenship: ********1234

Admin or verified authority sees:
Full number only after permission check
```

### Matching Logic for Missing Persons

The backend should compare new missing-person reports with existing found-person or sighting reports using:

* Name similarity
* Age range
* Gender
* Last seen location
* District
* Photo similarity
* Description similarity
* Date range

The system should calculate a match score.

```txt
Match Score = 
Name Similarity +
Age Similarity +
Location Similarity +
Image Similarity +
Description Similarity
```

Example:

```txt
80% or above  → Strong Match
60% to 79%    → Possible Match
Below 60%     → Weak Match
```


### Why Use Both Encryption and Hashing?

Encrypted value is used when authorized users need to view the original number.

Hash value is used for matching without exposing the real number.

```txt
Passport number: 123456789
Encrypted value: unreadable secure text
Hash value: fixed fingerprint used for searching
```

This allows the system to check whether a document already exists without revealing the original number.

## 🧠 Matching Engine

The Matching Engine is one of the most important parts of SahayogHub. It helps detect the most common or most likely match between reports.

### Matching Sources

```txt
Text data
Image data
Location data
Date and time
Document numbers
User-submitted descriptions
Category tags
```

### Matching Process

```txt
1. User submits a new report.
2. Backend normalizes the input.
3. System searches similar reports.
4. Matching score is calculated.
5. Top matching records are ranked.
6. User receives possible matches.
7. Admin or authority verifies the final match.
```

### Example Matching Score

```txt
Category match       = 20 points
Location match       = 20 points
Date range match     = 15 points
Text similarity      = 20 points
Image similarity     = 20 points
Verified document    = 25 points
```

The final score helps decide the strength of the match.

```txt
90+ points  → Very strong match
70-89       → Strong match
50-69       → Possible match
Below 50    → Low confidence match
```

## 🔐 Security Architecture

Since SahayogHub handles sensitive public and personal data, security is a core backend requirement.

### Authentication

The backend should use:

* JWT access token
* JWT refresh token
* Password hashing
* Email or phone verification
* Optional two-factor authentication for admins

### Authorization

The backend should apply role-based access control.

```txt
Citizen can create and view public reports.
Admin can approve and moderate reports.
Authority can access verified emergency and missing-person data.
Super Admin can manage system-wide settings.
```

### Sensitive Data Protection

The backend must protect:

* Citizenship numbers
* Passport numbers
* License numbers
* Phone numbers
* Home addresses
* Emergency location data
* Missing person details

Security techniques:

* Field-level encryption
* Hashed document numbers
* Permission checks
* Audit logs
* Secure file storage
* Private media URLs
* Rate limiting
* Input validation

## ⚙️ Background Tasks

Some tasks should not run directly inside the request-response cycle. They should run in the background using Celery and Redis.

Background jobs:

* Image processing
* Match score calculation
* Sending email/SMS alerts
* Generating notifications
* Cleaning duplicate reports
* Report expiration reminders
* Emergency escalation

```txt
Django API
   ↓
Celery Task Queue
   ↓
Redis Broker
   ↓
Worker processes matching, alerts, and image analysis
```
